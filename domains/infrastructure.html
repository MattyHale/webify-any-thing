Infrastructure

The foundational layer that determines whether everything else is possible or painful.

What is Expected

The EODI’s infrastructure is the core technical architecture underpinning the ingestion, management, and dissemination of Earth Observation data. It must seamlessly integrate many distributed components, data sources, and partner inputs into a unified whole that behaves as a single high-functioning platform.

In practice, this means supporting multi-constellation operations. Data from numerous satellite missions and providers are orchestrated under one system so that users experience one cohesive environment.

A unified platform removes silos and queue friction. Requests should trigger automated fulfilment that schedules collections, ingests results, standardizes outputs, and delivers them reliably. This shrinks latency, reduces operational overhead, and builds trust that the same system will perform under load and during critical windows.

Key Infrastructure Principles

Unified Data Integration

The EODI must bring together all relevant EO data sources including satellites, sensors, and archives under a common framework. Using a distributed system-of-systems architecture, the platform supports specialized processing for different data types while still presenting a unified interface. In practice, this means maintaining a centralized catalog or metadata repository that enables discovery across the entire holdings. Users should not need to know which satellite or archive a dataset came from. The infrastructure abstracts sources into one cohesive data lake. New suppliers or missions can be integrated without disrupting the overall system.

Self-Service and APIs

Every core function should be available via the user interface and stable APIs so users and external systems can place orders, track state, and retrieve results without tickets or emails.

Automated Fulfilment

Deterministic workflows should be triggered for tasking, acquisition monitoring, ingestion, preprocessing, and delivery, with recorded state transitions so every step is auditable.

Modular and Cloud-Native

Use independently deployable services for catalog, ordering, processing, delivery, and authentication. Components should scale in isolation and be updated without downtime.
